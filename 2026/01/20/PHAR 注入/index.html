PHAR 注入

2026-01-20
WEB安全
2.5k 词

PHAR 注入（也称为 PHAR 反序列化漏洞）是一种高级的 PHP 安全漏洞攻击技术。
它的核心概念是：利用 PHP 的 phar:// 伪协议在处理 PHAR 文件时，会自动反序列化其元数据（Metadata）的特性，从而在没有显式调用 unserialize() 函数的情况下，触发反序列化漏洞。

这通常被攻击者用来绕过代码审计，因为审计人员通常只盯着 unserialize() 找漏洞，而忽略了文件操作函数（如 file_exists）。

1. 什么是 PHAR 文件？

PHAR (PHP Archive) 是 PHP 的一种归档文件格式（类似于 Java 的 JAR 包）。它允许将多个 PHP 文件、资源和配置打包成一个文件。

一个标准的 PHAR 文件由四个部分组成：

  1. Stub (存根)： PHAR 文件的头部标识。必须以 __HALT_COMPILER(); 结尾。PHP 通过这个标识来识别这是不是一个 PHAR 文件。
  2. Manifest (清单)： 这是漏洞的关键。它包含了压缩文件的属性、权限等信息，以及用户自定义的元数据 (Metadata)。这些元数据是以 序列化 (Serialized) 的形式存储的。
  3. File Contents (文件内容)： 被压缩的实际文件。
  4. Signature (签名)： 文件完整性校验。

2. 漏洞原理

PHAR 注入的魔力在于：只要 PHP 通过 phar:// 协议解析一个 PHAR 文件，它就会强制将 Manifest 部分的 Metadata 进行反序列化。

这意味着，只要攻击者能控制任何一个文件操作函数的参数，并将其指向一个恶意的 PHAR 文件，就能触发反序列化攻击。

受影响的函数（不仅限于 include）： 这就大大扩展了攻击面。只要是涉及文件操作的函数，几乎都能触发：

  • file_exists()
  • is_dir(), is_file()
  • file_get_contents()
  • unlink()
  • scandir()
  • getimagesize()

3. 攻击流程

第一步：构造恶意 PHAR (生成 Payload)

攻击者在本地生成一个恶意的 PHAR 文件。这个文件会将一个带有恶意的对象序列化后写入 Metadata。

第二步：伪装与上传 (Bypass Upload Filters)

由于 PHAR 的识别只看 __HALT_COMPILER();，攻击者可以将 PHAR 后缀改为 .jpg、.gif，甚至在文件头加上图片标识（如 GIF89a）。

  • 结果： 服务器认为这是张图片，允许上传。

第三步：触发漏洞 (Trigger)

攻击者找到一个控制文件路径的地方（例如 is_file($_GET['path'])），并利用 phar:// 协议指向刚才上传的文件。

  • Payload: ?path=phar://./uploads/exploit.jpg/test.txt (或者直接指向文件本身，取决于具体函数)。

结果： PHP 解析 phar:// -> 读取 Metadata -> 自动反序列化 -> 触发 Evil 对象的 __destruct() -> RCE (远程代码执行)。

4. 如何防御？

  1. 禁用 phar:// 协议： 如果业务不需要，可以在 php.ini 中设置 phar.readonly = On（但这通常只阻止创建）。更好的方法是使用 stream_wrapper_unregister('phar'); 在代码中彻底注销该协议。

  2. 严格过滤输入路径： 检查用户输入的文件路径是否包含 phar://，或者直接限制只能包含 http:// 或纯相对路径。

  3. 文件上传检测： 不仅仅检测文件头和后缀，还应检测文件内容是否包含 __HALT_COMPILER 关键字。

  4. 业务逻辑隔离： 确保 file_exists 等函数的参数完全不可控，或者经过 basename() 处理。
