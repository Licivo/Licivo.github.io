什么是TemplatesImpl

TemplatesImpl 全称是 com.sun.org.apache.xalan.internal.xsltc.trax.TemplatesImpl。 在正常开发中，它是用来处理 XSLT 模板的（一种将 XML 转换为其他格式的技术）。为了提高性能，它会把 XSLT 转换成字节码并加载。

TemplatesImpl如何实现 RCE

TemplatesImpl 的攻击核心在于它内部的一个成员变量：_bytecodes。

关键属性

  • _bytecodes: 一个二维字节数组 (byte[][])。这里面存放的就是编译后的类文件字节码（即 .class 文件的二进制内容）。
  • _name: 必须不为 null，否则执行会提前退出。
  • _class: 缓存加载后的类。
  • _tfactory: 在某些 JDK 版本中需要被初始化（用于反射调用）。

漏洞触发链 (Call Chain)

攻击者通常会设法触发 TemplatesImpl.getOutputProperties() 方法。这个方法会触发一系列内部调用：

  1. getOutputProperties()：这是一个公共 Getter 方法（非常容易被 Fastjson 或 Jackson 自动触发）。
  2. newTransformer()：为了创建转换器，它需要获取模板实例。
  3. getTransletInstance()：它会检查类是否已经加载。
  4. defineTransletClasses()：核心步骤！ 它会获取自定义的 ClassLoader，并调用 defineClass 将 _bytecodes 里的字节数组转化为 Class 对象。
  5. newInstance()：类被加载后，紧接着会调用 newInstance() 创建实例。

只要你在 _bytecodes 里放入一段恶意代码（比如在静态代码块里写 Runtime.getRuntime().exec("calc")），当这个类被 newInstance() 实例化时，恶意代码就会立即执行。

为什么它在反序列化中这么火？

  1. 自带“加载器”：它不依赖外部的 .class 文件。恶意代码直接封装在序列化的 Payload 字节流中，随发随用。
  2. 绕过限制：即使服务器配置了严格的类路径检查，由于 _bytecodes 是在内存中直接定义并加载的，往往能绕过很多基于文件系统的防御。
  3. Getter 触发：很多流行的 JSON 库（如 Fastjson）在解析时会尝试调用所有的 getXXX 方法。TemplatesImpl 恰好有一个 getOutputProperties()，这使得它成为了 Fastjson 漏洞利用的标配。

类的初始化

为什么会触发静态代码

在 Java 中，一个类在被使用之前，必须经历 加载 (Loading) -> 链接 (Linking) -> 初始化 (Initialization) 三个阶段。

  • 加载：只是把磁盘或内存中的字节码读入 JVM（对应之前看到的 defineClass）。
  • 初始化：这是关键。JVM 规范规定，当程序发生以下行为（被称为“主动引用”）时，必须立即对类进行初始化：
    1. 使用 new 关键字创建实例。
    2. 调用类的反射方法（如 Class.forName() 或 newInstance()）。
    3. 访问类的静态变量或静态方法。
      当你调用 newInstance() 时： JVM 发现这个类虽然已经加载了，但还没有“初始化”，于是它会先执行类的初始化方法 <clinit>。这个 <clinit> 方法正是由编译器自动收集类中所有静态变量赋值动作和**静态语句块（static {}）合并产生的。

为什么会触发构造函数

触发机制：对象实例化（Instantiation）
newInstance() 的本质作用就是创建对象。它在语义上等同于调用 new T()。

  • 内存分配：JVM 首先在堆内存中为新对象分配空间。
  • 执行构造函数：为了确保对象进入可用状态，JVM 必须调用该类的构造函数（在字节码层面称为 <init> 方法）。
  • newInstance() 的特殊性：Class.newInstance() 专门用于调用该类的无参构造函数。

因此，只要写的代码在无参构造函数里，newInstance() 一旦成功创建对象，这些代码就必然会被执行。
